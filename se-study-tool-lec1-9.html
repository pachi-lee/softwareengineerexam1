Lecture 1 — Introduction to Software Engineering
01
Definition
Software Engineering
A disciplined approach for the development and maintenance of computer software using engineering principles to design, build, test, and maintain software systems.

Goal: deliver high-quality, efficient, user-focused software within time and cost constraints.
QualityEfficiencyUser-Focused
02
Core Process
SDLC
The Software Development Life Cycle is a structured process for planning, creating, testing, and deploying software.

5 Phases:
Maintenance → Requirements → Design → Implementation → Verification (cyclical)
5 PhasesCyclicalStructured
03
Requirements
User Stories
Short descriptions of a feature from the user's perspective.

Format: "As a [user], I want [goal] so that [reason]."

Paired with Acceptance Criteria (Given / When / Then) to define when a story is complete.
Given/When/ThenUser-Centric
04
Core Practices
3 Key Practices
1. Iterative Development — build in repeated cycles
2. Automated Testing — verify quality continuously
3. Version Control — track and manage code changes
IterativeTestingVersion Control
Lecture 2 — Requirements Engineering
01
Requirements
Functional vs Non-Functional
Functional: What the software DOES — observable behavior, features. Results are reproducible.

Non-Functional: How WELL it works — performance, security, scalability, load. Results may not be perfectly reproducible.
What = FunctionalHow Well = Non-Functional
02
Quality Checklist
INVEST
Independent — self-contained, any order
Negotiable — define what, not how
Valuable — product owner's need
Estimable — can be sized (S/M/L/XL)
Small — short to code and read
Testable — acceptance criteria cover all scenarios

Lecture 3 — Software Development Methodologies
01
Methodology
Waterfall
A linear, sequential model: Requirements → Design → Implementation → Verification → Maintenance.

Each phase must be fully complete before the next begins.

Use when: requirements are stable, small project, hardware-tied systems.
LinearSequentialStable Reqs
02
Methodology
Agile
Iterative, collaborative, flexible. Created in 2001.

4 Values: Individuals over Processes · Working Software over Documentation · Customer Collaboration over Contract · Responding to Change over Following a Plan
Iterative20014 Values
03
Framework
SCRUM
3 Roles: Product Owner, Scrum Master, Dev Team
3 Artifacts: Product Backlog, Sprint Backlog, Product Increment
5 Events: Sprint, Backlog Refinement, Daily Scrum, Sprint Review, Sprint Retrospective

SCRUM

Grooming takes ~5–10% of total sprint time.

An object is an instance of a class — a specific realization with its own attribute values.

Lecture 7 — Magic Methods, Inheritance & Interface
01
Inheritance
Single & Multiple
Single: one child inherits from one parent
Multiple: one child inherits from multiple parents

Syntax: class Child(Parent):
Child gets all parent methods automatically and can add its own.
is-a relationshipclass Child(Parent)
02
Interface
Abstract Base Classes
Python uses ABC (Abstract Base Classes) to enforce interface contracts.

from abc import ABC, abstractmethod

Subclasses MUST implement all @abstractmethod methods — otherwise a TypeError is raised on instantiation.
ABC Module@abstractmethodTypeError if not implemented
03
UML
Inheritance vs Realization
Inheritance (Generalization): solid line with hollow triangle ▷ pointing to parent

Interface (Realization): dashed line with hollow triangle --▷ pointing to interface
▷ = Inheritance--▷ = Implements
04
Pattern
Interface Example (Shape)
Shape (ABC) defines contract: all shapes must implement getArea():double.

Rectangle and Triangle provide concrete implementations.
This enforces consistent interface across all subclasses.
Contract PatternPolymorphism
Lecture 8 — Software Design & Databases
01
Design Principle
Cohesion vs Coupling
Cohesion
  • Focus of ONE module
  • Single purpose
  • Uniform interface
  • HIGH = good
Coupling
  • Between-module dep.
  • Degree of interaction
  • Hard to change solo
  • LOW = good
02
Abstraction
Hierarchy
Moving up = abstracting (more general). Moving down = refining (more specific).

automobile → car → sports car → Chevy Corvette → 1976 Stingray

"car is a refinement of automobile"
Up = AbstractDown = Refined
03
Database
Tables, PK, FK
Relational DB = tables (rows + columns).
PK uniquely identifies each row — duplicates cause exception.
FK is a column holding PK of another table, linking tables.
PK = Unique per rowFK = Links tables
04
Relationships
M:1 and M:M
Many-to-One: FK on the "many" side
Many-to-Many: Requires a junction table (two FKs, own PK)

ORM = Object-Relational Mapping (code instead of SQL). Examples: Django ORM, SQLAlchemy, Hibernate
Junction TableORM
